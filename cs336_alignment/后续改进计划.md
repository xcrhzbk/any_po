下面是一份**可以直接交给 codex 的统一实现计划**。目标是把这些方法做成**同一套训练框架下可切换的 7 个 trainer**，方便你统一画训练曲线、做同预算对比和项目讲解：

1. `VanillaGRPO`
2. `LambdaStepGRPO`
3. `ProgressGRPO`
4. `NegativeAwareProgressGRPO`
5. `OnlineGBPO`
6. `DAPO`
7. `GSPO`（第二阶段再做）

这 7 个方法的选择是有逻辑的：
GRPO 是基础 baseline；StepGRPO 代表“加步骤奖励”；Rewarding Progress 代表“步骤奖励应该奖励 progress 而不是局部表象”；NGRPO 代表“全负组要恢复学习信号”；GBPO 是 bounded-update 家族；DAPO 是工程上非常强的 GRPO recipe；GSPO 则从 token-level ratio 转向 sequence-level ratio，是更后期的目标函数级改动。([arxiv.org](https://arxiv.org/abs/2402.03300?utm_source=chatgpt.com) ([arXiv][1]))

# 一、项目总目标

做一套**统一训练与评测框架**，比较以下问题：

* 稀疏 final reward 的 GRPO，和 step-level reward 的 GRPO，谁收敛更快。
* “普通 step reward”和“progress-style step reward”相比，谁更稳定、谁更有效。([arxiv.org](https://arxiv.org/abs/2410.08146?utm_source=chatgpt.com))
* 在 all-negative groups 上，额外的 step-level progress 是否能恢复学习信号。([arxiv.org](https://arxiv.org/abs/2509.18851?utm_source=chatgpt.com))
* bounded update（GBPO）和 decoupled clip/dynamic sampling（DAPO）能否进一步提升训练稳定性。([arxiv.org](https://arxiv.org/abs/2508.20900) ([arXiv][2]))
* sequence-level ratio（GSPO）在同样数据和预算下，是否比 token-level ratio 更稳。([arxiv.org](https://arxiv.org/abs/2507.18071))

---

# 二、统一设计原则

## 1. 所有方法共用一套 rollout / parser / reward 接口

不要为每个方法写一套独立工程。
应当有统一的：

* rollout 生成
* step parser
* final reward
* step reward
* advantage builder
* loss builder
* logging

这样后面你做曲线对比才公平。

## 2. 除核心算法外，其他训练设定必须固定

以下设置在所有 trainer 间保持一致：

* base model
* tokenizer
* dataset split
* prompt template
* max_new_tokens
* batch size
* group size
* optimizer
* lr schedule
* kl coefficient
* total updates

## 3. 曲线比较以“相同算力 / 相同 token 预算 / 相同 rollout 数”为主

不然 DAPO、GSPO、GBPO 这类方法容易被误解成只是更吃算力。

# 四、统一公共基础设施

这部分必须先做完，后面所有 trainer 共用。

## 4.1 结构化输出格式

要求模型输出可切分步骤，例如：

```text
<reasoning>
Step 1: ...
<step_end>
Step 2: ...
<step_end>
...
</reasoning>
<answer>
...
</answer>
```

原因是 LambdaStepGRPO、ProgressGRPO、NegativeAwareProgressGRPO 都依赖 step segmentation；Thought-ICS/self-localization 一类工作也说明，结构化 discrete thought steps 更利于错误定位和局部修复。([arxiv.org](https://arxiv.org/abs/2602.02416))

## 4.2 统一 rollout 数据结构

```python
from dataclasses import dataclass
from typing import List, Tuple, Optional, Dict, Any

@dataclass
class ParsedRollout:
    prompt: str
    response_text: str
    answer_text: str

    steps: List[str]
    step_token_spans: List[Tuple[int, int]]

    final_reward: float = 0.0

    prefix_scores: Optional[List[float]] = None
    step_rewards: Optional[List[float]] = None
    step_returns: Optional[List[float]] = None
    step_advantages: Optional[List[float]] = None

    meta: Optional[Dict[str, Any]] = None
```

## 4.3 通用 parser

统一接口：

```python
class RolloutParser:
    def parse(self, prompt: str, response_text: str, tokenizer) -> ParsedRollout:
        ...
```

必须保证：

* `len(steps) == len(step_token_spans)`
* span 不重叠
* 空 step 被过滤
* answer 稳定可抽取

## 4.4 通用 token-level loss 接口

所有 trainer 最后都输出 `token_adv` 给统一 loss：

```python
def grpo_token_loss(
    logp_new,
    logp_old,
    logp_ref,
    token_adv,
    token_mask,
    clip_eps=0.2,
    beta_kl=0.01,
):
    ...
```

GSPO 例外，它会改 ratio 定义，但仍建议复用统一 trainer 框架，只替换 `loss_builder`。([arxiv.org](https://arxiv.org/abs/2507.18071))

# 五、七个方法的实现计划

---

## A. VanillaGRPO

## 目标

复现标准 GRPO baseline。GRPO 最早在 DeepSeekMath 中提出，特点是组内相对奖励、不用显式 critic。([arxiv.org](https://arxiv.org/abs/2402.03300?utm_source=chatgpt.com))

## 逻辑

1. 每个 prompt 采样 `G` 条 rollout
2. 计算 sequence-level final reward
3. 组内标准化，得到 `seq_adv`
4. 把 `seq_adv` 平铺到该 rollout 全部 token

## 需要实现

* `VanillaGRPOTrainer`
* `normalize_within_group(final_rewards)`
* `expand_seq_adv_to_tokens(seq_adv, token_mask)`

## 关键日志

* `train/final_reward_mean`
* `train/final_reward_std`
* `train/all_negative_group_ratio`
* `train/all_positive_group_ratio`
* `train/avg_response_len`
* `train/avg_step_count`

---

## B. LambdaStepGRPO

StepGRPO 已经证明 step-wise dense reward 可有效缓解稀疏奖励；这里再叠一个 λ-return 做更细粒度 credit assignment。([arxiv.org](https://arxiv.org/abs/2503.12937?utm_source=chatgpt.com))

## 目标

把 sequence-level advantage 升级成 step-level advantage。

## 核心思路

每一步一个即时奖励 `r_step[j]`，再用 λ-return 计算该步回报：

* `R_step[j] = r_step[j] + gamma * lam * R_step[j+1]`
* 初始尾项来自 `final_reward`

## Step reward 第一版实现

先实现一个便宜的 `PrefixScorer`：

```python
class PrefixScorer:
    def score(self, prompt: str, prefix_steps: list[str]) -> float:
        ...
```

第一版允许 heuristic：

* 非空
* 非重复
* 格式合理
* 与题目实体有重叠
* 不明显胡言乱语

对每个 prefix 得到 `V_j`，再定义：

* `r_step[j] = V_j - V_{j-1}`

## 需要实现

* `PrefixScorer`
* `compute_step_returns(step_rewards, final_reward, gamma=1.0, lam=0.95)`
* `normalize_step_returns(step_returns, step_mask)`
* `expand_step_adv_to_tokens(step_adv, step_token_spans, T)`

## 训练循环

1. rollout + parse
2. final reward
3. 逐步算 prefix score
4. 差分得到 `step_rewards`
5. 算 `step_returns`
6. 对 group 内每个 step index 单独归一化
7. 展开为 token advantage
8. 进统一 GRPO loss

## 关键日志

* `train/step_reward_mean`
* `train/step_return_mean`
* `train/prefix_score_last_mean`
* `train/avg_step_adv_abs`

## 目的

这是**强 baseline**，不要把它当主创新。

---

## C. ProgressGRPO

Rewarding Progress 的关键 insight 是：过程奖励应该是“该步让未来成功概率提升了多少”，而不是局部格式或表象得分。([arxiv.org](https://arxiv.org/abs/2410.08146?utm_source=chatgpt.com))

## 目标

把 step reward 从“prefix quality delta”升级成“future success delta”。

## 核心定义

对每个前缀：

* `P_j = Prob(correct final answer | prompt, prefix_j)`

然后定义：

* `r_step[j] = P_j - P_{j-1}`

## 新增模块：ProgressScorer

```python
class ProgressScorer:
    def score_prefixes(self, prompt: str, steps: list[str]) -> list[float]:
        ...
```

输出 `[P_1, ..., P_K]`

## ProgressScorer 的训练方式

### 阶段 A：离线构造 prefix-progress 数据

实现脚本：

```python
build_prefix_progress_dataset.py
```

对一批 rollout，只选少量关键 prefix（建议每条 3 个），每个 prefix 做 4 到 8 次 continuation sampling，统计最终正确率，形成：

```python
{
  "prompt": ...,
  "prefix_text": ...,
  "prefix_steps": [...],
  "success_rate": P_j
}
```

### 阶段 B：训练小 scorer

输入：prompt + prefix
输出：`P_j`
loss：MSE 或 BCE

## 训练时逻辑

1. rollout + parse
2. final reward
3. `ProgressScorer.score_prefixes()` 得到 `[P_j]`
4. 差分得到 `step_rewards`
5. 算 `step_returns`
6. step-wise group normalize
7. 展开到 token
8. 统一 GRPO loss

## 关键日志

* `train/progress_prefix_mean`
* `train/progress_delta_mean`
* `train/progress_delta_std`
* `val/progress_mse`
* `val/progress_calibration_error`

## 验收标准

至少比：

* VanillaGRPO
* LambdaStepGRPO
  更好，否则主方法不成立。

## D. NegativeAwareProgressGRPO

NGRPO 的核心问题意识是：GRPO 在 homogeneous groups，尤其全负组上，group-relative advantage 会失效。你的版本用 step-level progress 去救 all-negative groups。([arxiv.org](https://arxiv.org/abs/2509.18851?utm_source=chatgpt.com))

## 目标

在 all-negative groups 中恢复学习信号。

## all-negative 判定

对每个 prompt 的 group：

* 如果 `sum(final_rewards) == 0`，则 `group_all_negative = True`

## 核心思路

混合 sequence-level advantage 和 step-level advantage：

* 普通组：`A_total = A_seq + alpha_normal * A_step`
* 全负组：`A_total = A_seq + alpha_neg * A_step`

建议默认：

* `alpha_normal = 0.5`
* `alpha_neg = 1.5`

## 可选增强：rescue score

支持但默认关闭：

* `rescue_score = sum(max(r_step[j], 0))`

如果启用，则 all-negative 组可用 rescue score 代替 final reward 再做 group normalization。

## 训练逻辑

1. 先完整跑 ProgressGRPO
2. 额外判断哪些 group 是 all-negative
3. 用不同 alpha 混合 sequence 和 step advantage

## 必须记录

* `train/all_negative_group_ratio`
* `train/all_negative_avg_step_adv`
* `train/non_negative_avg_step_adv`
* `train/alpha_neg`
* `train/alpha_normal`

## 验收标准

至少验证：

* 在 all-negative-heavy 阶段，优于 ProgressGRPO
* 不是单纯更大 step reward，而是确实提高有效更新

---

## E. OnlineGBPO

OneRec-V2 提到 GBPO；后续工作把它概括为一种 bounded policy optimization 思路，重点是对更新边界做更稳的控制，尤其控制负样本或极端 ratio 的更新。([arxiv.org](https://arxiv.org/abs/2508.20900))

## 目标

做一个独立 trainer，测试 bounded update 是否能提升在线 GRPO 的稳定性。

## 重要说明

这里不要求你严格复刻推荐场景里的 BCE 推导。
第一版只实现 **GBPO-style sign-aware bounded update**：

* advantage > 0 时，允许更大更新
* advantage < 0 时，限制更严格

## 建议定义

对 ratio：

* 正优势样本用 `(pos_low, pos_high)`
* 负优势样本用 `(neg_low, neg_high)`

推荐默认：

* `pos_low = 0.2`
* `pos_high = 0.28`
* `neg_low = 0.05`
* `neg_high = 0.10`

loss 形式：

```python
if adv >= 0:
    ratio_eff = clamp(ratio, 1 - pos_low, 1 + pos_high)
else:
    ratio_eff = clamp(ratio, 1 - neg_low, 1 + neg_high)
loss = -adv * ratio_eff
```

## 需要实现

* `gbpo_bound.py`
* `OnlineGBPOTrainer`

## 这个 trainer 用哪个 reward

第一阶段先只配 **sequence-level reward**，也就是：

* 结构上接近 VanillaGRPO
* loss 上改成 GBPO-style

### 第二阶段可选

支持把 GBPO bound 加到 ProgressGRPO 上，但那是扩展实验，不是主干实现。

## 关键日志

* `train/pos_ratio_clip_hit_rate`
* `train/neg_ratio_clip_hit_rate`
* `train/avg_abs_ratio`
* `train/avg_abs_adv`
* `train/neg_adv_grad_norm`

## 验收标准

至少看：

* loss 曲线更稳
* gradient norm 更稳
* 不弱于 VanillaGRPO

## F. DAPOTrainer

DAPO 的公开 recipe 很明确：重点不是单一 loss，而是 decoupled clip、dynamic sampling with group filtering、flexible loss aggregation、以及 overlong reward shaping。([arxiv.org](https://arxiv.org/abs/2503.14476) ([verl][3]))

## 目标

做一个独立 trainer，尽量接近论文 recipe，但先聚焦最关键的 3 个模块：

1. `decoupled clip`
2. `dynamic sampling / group filtering`
3. `flexible loss aggregation`

### overlong reward shaping

支持接口，但先作为可选项，不是第一阶段重点。

---

## F1. Decoupled Clip

实现两组 clip：

* `clip_ratio_low`
* `clip_ratio_high`

默认：

* `clip_ratio_low = 0.2`
* `clip_ratio_high = 0.28`

loss 里不再用对称 clip。

这点和 GBPO 很接近，但 DAPO 默认不做 sign-aware，再加上 dynamic sampling 才构成完整 DAPO。([verl.readthedocs.io](https://verl.readthedocs.io/en/latest/algo/dapo.html))

---

## F2. Dynamic Sampling with Group Filtering

实现 group filter：

* 丢掉 all-0 group
* 丢掉 all-1 group
* 如果有效 group 不够，继续采样直到攒够训练 batch

需要实现：

* `sampling/dapo_group_filter.py`

配置项：

* `enable_group_filter: true`
* `filter_all_zero: true`
* `filter_all_one: true`
* `target_valid_groups_per_batch`

这会和 NegativeAwareProgressGRPO 构成很好的对比：

* DAPO：过滤 homogeneous group
* NegativeAware：利用 all-negative group

---

## F3. Flexible Loss Aggregation

支持 3 种：

* `token_mean`
* `seq_mean_token_sum`
* `seq_mean_token_mean`

做成 config 开关。

这对长输出场景很重要，因为不同 aggregation 会影响长链奖励的稀释方式。([verl.readthedocs.io](https://verl.readthedocs.io/en/latest/algo/dapo.html))

---

## F4. Overlong Reward Shaping（可选）

预留接口：

* 如果 response 过长，给长度惩罚或超长 shaping

先不做复杂版本，只要能在 config 打开/关闭。

---

## F5. DAPOTrainer 训练逻辑

和 VanillaGRPO 大体一样，但：

* loss builder 换成 decoupled clip
* sampler 增加 group filtering
* aggregation 可配置

## 关键日志

* `train/valid_group_ratio`
* `train/resample_count`
* `train/clip_low_hit_rate`
* `train/clip_high_hit_rate`
* `train/loss_agg_mode`

## 验收标准

至少比较：

* VanillaGRPO vs DAPOTrainer
* OnlineGBPO vs DAPOTrainer

## G. GSPOTrainer（第二阶段）

GSPO 的核心是把 importance ratio 从 token-level 改到 sequence-level，并做 sequence-level clipping、rewarding 和 optimization。([arxiv.org](https://arxiv.org/abs/2507.18071))

## 为什么放第二阶段

因为 GSPO 是目标函数级别的改动，会和 step-level advantage 设计发生冲突。
所以第一阶段先只做一个**纯 GSPO baseline trainer**，不要一上来和 progress reward 混。

## 目标

复现一个 sequence-level ratio 版本，作为后期稳定性对照。

## 设计原则

* 先只配 sequence-level reward
* ratio 用 sequence log-likelihood 计算
* clipping 用 sequence-level
* loss aggregation 默认 sequence-level

## 需要实现

* `gspo_loss.py`
* `GSPOTrainer`

## 注意

GSPO 第一版不要接：

* LambdaStepGRPO
* ProgressGRPO
* NegativeAware

先保持纯净 baseline。

## 关键日志

* `train/seq_ratio_mean`
* `train/seq_ratio_std`
* `train/seq_clip_hit_rate`
* `train/seq_loss_mean`

## 验收标准

作为后期 baseline，对比：

* VanillaGRPO
* DAPOTrainer
* GSPOTrainer

# 十一、给 codex 的一句话任务定义

> 在现有 GRPO 训练代码上，统一实现 7 个可切换 trainer：VanillaGRPO、LambdaStepGRPO、ProgressGRPO、NegativeAwareProgressGRPO、OnlineGBPO、DAPO、GSPO。所有 trainer 共用同一 rollout、parser、reward、logging 和评测体系；除核心 loss / reward / sampling 差异外，训练设定完全一致，方便公平比较训练曲线与最终性能。

# 十二、我的建议：怎么避免工程发散

最重要的一条：

**GSPO 第二阶段再做。**
因为 GSPO 把 ratio 定义改到 sequence-level，它和你前面做的 step-level credit assignment 不是一层改动；先把前 6 个 trainer 跑通，再做 GSPO，对项目最稳。GSPO 的原论文也明确把 sequence-level ratio 作为其核心区别。([arxiv.org](https://arxiv.org/abs/2507.18071))
